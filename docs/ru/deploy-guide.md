# Deploy-квест: Как мы настраивали автоматический деплой Smarthome через GitHub Actions

Когда мы начинали настраивать `deploy.yml`, казалось, что достаточно просто добавить workflow, и GitHub сам всё сделает. Реальность оказалась строже: права доступа, пустые коммиты и «битые» пути. Вот наш «список граблей», на которые мы наступили, чтобы вы могли их перешагнуть.

## Проблема 1: GitHub Actions «без прав»
Самая первая ошибка — `403 Forbidden`. GitHub Actions по умолчанию *не имеет* права менять содержимое репозитория.

**Решение:** В `deploy.yml` нужно явно прописать права:
```yaml
permissions:
  contents: write
```

## Проблема 2: Анонимный коммит
Runner внутри GitHub Actions — это «никто». Если скрипт пытается сделать коммит (а `mkdocs gh-deploy` именно это и делает), он не знает, чьё имя подставить в `git commit`.

**Решение:** Добавьте шаг инициализации Git перед деплоем:
```yaml
- run: |
    git config --global user.name "GitHub Action"
    git config --global user.email "action@github.com"
```

## Проблема 3: Ловушка абсолютных путей
Когда вы пишете `[Статья](/gog-cli)` в Markdown, на локальном сервере это работает идеально. Но при деплое в подпапку (например, `trifonovhome.github.io/smarthome/`), абсолютный корень `/` ведет в никуда.

**Решение:** Используйте относительные пути. Мы перевели всю внутреннюю перелинковку на относительные ссылки, чтобы сайт «подхватывал» базовый URL автоматически.

## Проблема 4: DNS и CNAME
Настройка домена `trifonov.me` — это не только про `CNAME` в настройках репозитория. Если GitHub не видит файл `CNAME` в корне ветки `gh-pages`, он будет сбрасывать настройки домена при каждом деплое.

**Решение:** Создайте файл `CNAME` с текстом `trifonov.me` в папке `docs/`, чтобы он всегда попадал в итоговый билд.

## Чек-лист для запуска
1. Дать права `contents: write` в workflow.
2. Настроить `git config` в CI.
3. Забыть про абсолютные пути (`/`) в ссылках.
4. Проверить файл `CNAME` в билде.
